# Color Queens 产品需求文档

最后更新：2026-07-05  
适用分支：`codex-iteration`  
当前阶段：Godot MVP

## 1. 文档目的

本文档用于记录 Color Queens 当前产品范围、功能模块、功能逻辑、实现原理和回归测试要求。后续每次功能迭代都必须同步更新本文档，并在正式发布前按本文档执行测试回归。

本文档是项目的产品事实源：

- 新增功能时，先补充或更新对应模块说明。
- 修改玩法、经济、界面、关卡或提示逻辑时，同步更新受影响模块。
- 发布前，以“回归测试清单”为准逐项验证。

## 2. 产品概述

Color Queens 是一款竖屏移动端休闲逻辑谜题。玩家在 6x6 彩色区域棋盘中放置皇冠，需要同时满足四条规则：

- 每一行只能有一个皇冠。
- 每一列只能有一个皇冠。
- 每个颜色区域只能有一个皇冠。
- 皇冠不能八方向相邻。

当前产品参考 Royal Match / Magic Sort 的休闲闯关外壳，但核心玩法保持 Queen / Star Battle 类逻辑谜题，不改为倒水、三消或排序玩法。

## 3. 目标用户与体验方向

目标用户是偏休闲的逻辑谜题玩家。产品体验目标：

- 上手轻：通过首页、关卡、提示和即时纠错降低理解成本。
- 逻辑清楚：提示要解释“为什么”，不能只告诉玩家答案。
- 节奏短：单关 6x6，适合碎片化游玩。
- 可持续迭代：关卡、提示、经济和主界面可以逐步扩展。

## 4. 当前版本范围

已实现：

- 休闲闯关首页。
- 6x6 Color Queens 棋盘玩法。
- 50 个默认关卡。
- 难度标记：新手、普通、困难、专家。
- 无猜解题路径数据：`logicStatus`、`hintSteps`、`solveSteps`。
- 教学提示：单候选、候选锁定、成组锁定、反证排除、直接冲突排除。
- 金币、提示次数、通关奖励。
- 本地存档。
- 内置关卡编辑器。
- Godot 冒烟测试和编辑器冒烟测试。

未完成或占位：

- 生命系统目前只展示固定数值。
- 排行榜、活动、队伍、商店、宝箱为占位入口。
- 广告位为 Demo Placeholder。
- 关卡编辑器偏开发工具，尚未做生产级权限和发布流程。

## 5. 信息架构

主要页面：

- 首页：资源展示、章节进度、主视觉、开始关卡、任务、活动/排行榜/编辑器等入口。
- 关卡页：关卡选择、棋盘、规则提示、进度、撤销、清除、提示、广告占位。
- 通关弹窗：通关反馈、金币奖励、下一关、重玩本关。
- 帮助弹窗：基础规则说明。
- 关卡编辑器：编辑颜色区域和答案点位。

核心数据：

- `data/levels.json`：关卡配置、答案、难度、提示路径和无猜解题路径。
- `user://color_queens_save.json`：本地存档。

## 6. 模块需求

### 6.1 首页模块

用户目标：明确当前在 Color Queens 主界面，看到资源、进度和主操作入口，一键进入当前关卡。

功能逻辑：

- 首屏显示 sky/garden/lake/path 风格背景和 Color Queens 主视觉。
- 顶部资源栏显示金币、生命、完成星数和设置按钮。
- 侧边浮动入口包括每日奖励、宝箱、编辑器、活动、排行榜。
- 中下部显示“开始关卡”和“任务”按钮。
- 底部导航显示星、杯、主页、队、设等入口，其中主页为当前选中状态。
- 首页进度根据当前关卡和已完成关卡数更新。

当前实现：

- UI 由 `scripts/main.gd` 中 `_build_home_screen()` 及其子函数动态构建。
- 金币、生命、星数使用 Label 展示。
- 每日奖励按钮直接增加金币并保存。
- 活动、排行榜、任务、宝箱等按钮目前触发 toast，占位不进入真实系统。

回归测试重点：

- 启动后默认进入首页。
- 点击开始关卡进入关卡页。
- 点击每日奖励金币增加并保存。
- 占位入口不会导致报错或页面卡死。

### 6.2 关卡页模块

用户目标：完成当前关卡，清楚知道关卡进度、剩余操作和规则反馈。

功能逻辑：

- 顶部栏包含返回首页、生命、关卡下拉、编辑器、设置。
- 关卡标题只显示关卡编号，不显示关卡名称。
- 进度条显示已放皇冠数 / 目标皇冠数。
- 教练提示区显示关卡教程或当前提示解释。
- 棋盘占据主要空间，底部为撤销、清除、提示按钮。
- 广告位保留在底部。

当前实现：

- UI 由 `scripts/main.gd` 中 `_build_game_screen()` 及其子函数构建。
- 关卡切换使用 `OptionButton`，当前属于调试/开发便利入口。
- 进度来自 `_piece_positions()` 统计。
- 冲突状态由 `_validate_and_update()` 更新。

回归测试重点：

- 页面在移动端宽度下按钮和棋盘不溢出。
- 关卡页不显示金币和排行榜。
- 关卡标题和下拉只显示关卡编号。
- 切换关卡会重置棋盘并加载对应数据。

### 6.3 棋盘交互模块

用户目标：通过点击格子标记排除或放置皇冠，直观看到颜色区域、皇冠、X、冲突和提示高亮。

功能逻辑：

- 单格状态循环：空白 -> 排除标记 X -> 皇冠 -> 空白。
- 棋盘支持鼠标点击和触屏点击。
- 有冲突时冲突格变红。
- 提示时显示观察范围、候选、排除和建议格。
- 操作后提供轻量动画反馈。

当前实现：

- 交互状态由 `scripts/main.gd` 中 `cell_states` 管理。
- 棋盘绘制由 `scripts/game_board.gd` 自绘完成。
- `GameBoard` 通过 `cell_pressed(row, col)` 信号把点击传回主逻辑。
- 高亮类型包括 `unit`、`candidate`、`exclude`、`place`。
- `play_cell_feedback()` 和 `play_guide_feedback()` 提供格子反馈动画。

回归测试重点：

- 第一次点击为空格标记 X。
- 第二次点击同格放皇冠。
- 第三次点击同格恢复空白。
- X、皇冠、高亮和冲突图形可见且不遮挡棋盘。
- 棋盘在不同窗口尺寸下保持正方形比例。

### 6.4 规则与冲突模块

用户目标：放错时能及时知道哪里违反规则。

功能逻辑：

- 任意两个皇冠如果同行、同列、同颜色区域或八方向相邻，则视为冲突。
- 开启即时纠错时，冲突格高亮。
- 设置按钮可切换即时纠错。

当前实现：

- `_find_conflicts()` 遍历当前皇冠位置并检查四类冲突。
- `_validate_and_update()` 负责刷新错误状态、进度和教练提示。
- `immediate_errors` 控制是否显示即时错误。

回归测试重点：

- 同行、同列、同区域、相邻两个皇冠均显示冲突。
- 关闭即时纠错后不显示红色冲突，但通关仍要求无冲突。

### 6.5 提示模块

用户目标：获得当前最值得判断的一步，并理解为什么可以选择或排除。

功能逻辑：

- 提示不直接替玩家放置皇冠。
- 提示消耗免费次数；免费次数用完后消耗金币。
- 免费提示初始为 3 次。
- 金币提示成本为 5。
- 提示给出文字解释和棋盘高亮。
- 提示应优先给出可以推进棋盘的逻辑判断。

当前提示策略顺序：

1. 单元唯一候选：某行、列或颜色区域只剩一个合法候选。
2. 候选锁定：某单元候选全部落在另一单元内，可排除另一单元其它候选。
3. 成组锁定：多个单元共同锁定一组行、列或颜色区域，可排除组外候选。
4. 反证排除：如果某格放皇冠，会导致某个单元没有任何可放位置。
5. 直接冲突排除：已有皇冠导致某格必然冲突。
6. 候选关注：没有确定推进时，提示值得比较的候选范围。

当前实现：

- `_use_hint()` 执行提示消耗、设置高亮和更新教练文本。
- `_build_best_next_hint()` 负责选择当前最佳提示。
- 相关策略函数集中在 `scripts/main.gd` 的提示逻辑区。
- 区域名称由 `REGION_COLOR_NAMES` 转换为可见颜色名，例如黄色区域、绿色区域。
- 关卡数据中的 `solveSteps` 记录无猜路径，但当前运行时提示主要按当前棋盘动态计算。

回归测试重点：

- 点击提示不会自动放皇冠。
- 提示会消耗 1 次免费提示或 5 金币。
- 金币和免费提示不足时给出 toast。
- 提示文案必须解释原因，不只说“这里最优”。
- 提示高亮至少包含一个目标格。
- 对 50 关至少能从数据层证明唯一解和 no-guess 路径存在。

### 6.6 关卡数据模块

用户目标：获得稳定、可解、难度递进的关卡内容。

功能逻辑：

- 默认包含 50 个 6x6 关卡。
- 每关包含唯一答案。
- 每关必须标记难度。
- 每关必须标记 `logicStatus: no_guess`。
- 每关必须包含 `hintSteps` 和 `solveSteps`。

当前实现：

- 数据文件：`data/levels.json`。
- 加载器：`scripts/level_store.gd`。
- 基础校验包括行列尺寸、区域行列数和答案数量。
- 更完整的唯一解与路径校验目前由 `tests/smoke_test.gd` 覆盖。

回归测试重点：

- `levels.size() >= 50`。
- 每关为 6x6。
- 每关答案满足行、列、区域唯一和不相邻。
- 每关唯一解。
- 每关存在难度、`logicStatus`、`hintSteps`、`solveSteps`。

### 6.7 通关与奖励模块

用户目标：完成关卡后获得明确反馈，并能进入下一关或重玩。

功能逻辑：

- 当皇冠数量等于目标数且无冲突时通关。
- 首次完成关卡奖励金币。
- 通关弹窗显示皇冠、完成标题、金币奖励、下一关和重玩按钮。
- 下一关按钮进入下一关；最后一关则回到首页。
- 重玩本关清空棋盘并重新开始。

当前实现：

- `_complete_level()` 处理完成状态、奖励、保存和弹窗。
- `WIN_REWARD` 当前为 10。
- `_next_level()` 和 `_replay_level()` 处理弹窗按钮。
- `completion_overlay` 为全屏遮罩弹窗。

回归测试重点：

- 正确解完成后弹出通关弹窗。
- 首次通关金币增加 10。
- 已完成关卡重复通关不重复奖励。
- 下一关按钮尺寸足够明显且可点击。
- 重玩本关清空棋盘。

### 6.8 存档模块

用户目标：退出后能恢复当前进度。

功能逻辑：

- 保存当前关卡、棋盘状态、金币、提示次数、已完成关卡和完成状态。
- 启动时自动读取存档。
- 存档版本支持兼容迁移。

当前实现：

- 存档路径：`user://color_queens_save.json`。
- `_load_save()` 加载并兼容旧版本。
- `_save_game()` 在关键状态变化后写入。
- `SAVE_VERSION` 当前为 2。

回归测试重点：

- 关卡切换、点击格子、使用提示、通关后会保存。
- 重新启动后恢复最近关卡和棋盘状态。
- 旧版本存档不会导致启动失败。

### 6.9 帮助与设置模块

用户目标：快速查看基础玩法，控制是否即时显示错误。

功能逻辑：

- 帮助弹窗说明四条规则和点击循环。
- 设置按钮切换即时纠错。

当前实现：

- `_build_help_dialog()` 创建帮助弹窗。
- `_on_help()` 展示帮助。
- `_on_settings()` 切换 `immediate_errors` 并重新校验棋盘。

回归测试重点：

- 帮助弹窗可打开和关闭。
- 设置切换后 toast 提示状态。
- 即时纠错开关影响红色冲突显示。

### 6.10 关卡编辑器模块

用户目标：开发期快速编辑关卡区域和答案。

功能逻辑：

- 可加载已有关卡。
- 可切换区域涂色和答案模式。
- 可调整关卡名称、提示文字、颜色区域和答案点位。

当前实现：

- 场景：`scenes/level_editor.tscn`。
- 脚本：`scripts/level_editor.gd`。
- 入口在首页侧边按钮和关卡页顶部按钮。
- 目前编辑器偏开发工具，不作为正式玩家功能。

回归测试重点：

- 编辑器场景可加载。
- 区域模式可涂色。
- 答案模式可切换点位。
- 从主界面入口进入不会报错。

## 8. 发布前回归测试清单

每次正式发布前必须完成：

1. 运行核心冒烟测试。
2. 运行编辑器冒烟测试。
3. 手动验证首页主要入口。
4. 手动验证关卡页在目标移动端尺寸下不溢出。
5. 手动完成至少 1 个新手关、1 个普通关、1 个困难或专家关。
6. 手动验证提示不会直接放皇冠，并且文案解释原因。
7. 手动验证通关弹窗、下一关、重玩本关。
8. 如果修改了 `data/levels.json`，必须确认每关仍唯一解且 `logicStatus` 为 `no_guess`。
9. 如果修改了存档结构，必须验证旧存档兼容。
10. 更新本文档对应模块。

## 10. 迭代更新规则

每次迭代按以下顺序执行：

1. 确认当前分支是 `codex-iteration` 或明确指定的开发分支。
2. 阅读本文档，确认要改的模块和影响范围。
3. 修改功能实现。
4. 同步更新本文档：新功能加入对应模块，行为变化更新功能逻辑，参数变化更新关键参数，新风险加入回归测试清单。
5. 运行自动测试。
6. 按影响范围做手动回归。
7. 提交代码。

## 11. 后续建议

优先补齐：

- 把编辑器从玩家入口中隐藏或加开发开关。
- 给提示系统增加可视化分层，例如“观察范围”“可排除原因”“建议操作”。
- 增加手机真机或 Godot 移动预览回归步骤。
- 将活动、排行榜、宝箱、任务等占位入口拆成明确版本规划。
